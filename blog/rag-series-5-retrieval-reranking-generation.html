RAG Series Search & AI Part 5

RAG Explained, Part 5: Retrieval, Reranking & Generation

June 9, 2026 15 min read Dhaval Shah

This is the finale. We can already embed a question and pull the nearest chunks from our index (Parts 3 and 4). Now we turn that raw retrieval into a polished, trustworthy answer: how many chunks to fetch, how to combine vector and keyword search, how reranking sharpens precision, and how the model finally generates a grounded reply with citations. Then we'll cover the part everyone skips — how to know if it's actually any good.

Quick recap of the live query path from Part 1: embed the query → search → augment the prompt → generate. This article expands "search → augment → generate" into the techniques that make answers reliable.

Step 1 — Choose how many chunks: top-k

The vector index can return its closest matches ranked by similarity. Top-k is simply how many you keep. It's a balancing act:

k too small (e.g. 1)k too large (e.g. 20)
If the single chunk misses part of the answer, the model has nothing to fall back on.You drag in loosely related chunks that distract the model and cost more tokens.
Risk: incomplete answers.Risk: vague answers, higher cost, slower responses.

A typical starting point is k = 3 to 5. Enough to cover an answer that spans a couple of passages, few enough to stay focused. Like chunk size, you tune it by watching real questions.

Step 2 — Search smarter: hybrid + filters

Pure vector search is excellent at meaning but, as we noted in Part 3, it can be fuzzy on exact tokens — order numbers, product codes, error IDs, proper names. Keyword search is the opposite. So the strongest systems run hybrid search: both at once, then merge the results.

Hybrid search — two retrievers, merged
User question"return a faulty laptop, order #A-1138?"
Meaning
Vector searchFinds the refund/returns policy
Exact words
Keyword searchMatches the exact order "#A-1138"
Merged resultsBest of both, combined into one ranked list

Hybrid search captures meaning and exact tokens — the order number and the policy both surface.

On top of search you add metadata filters — the metadata we attached to every chunk back in Part 2. Filters narrow the candidate pool before ranking, which improves both accuracy and security:

  • Freshness: only chunks where status = current, so you never answer from a superseded policy.
  • Scope: only department = HR, or language = en, or region = EU.
  • Security: only chunks the signed-in user is permitted to see — the most important filter of all.
Security trimming is non-negotiable. Apply the user's permissions as a filter at retrieval time. Without it, RAG can cheerfully quote a confidential document to someone who should never see it. Retrieval must respect the same access rules as the source system.

Step 3 — Reranking: a smarter second opinion

Vector search is fast but a little rough — it compares the query and each chunk independently. A reranker is a slower, more accurate model that looks at the query and a candidate chunk together and scores how well that chunk actually answers this question. The trick is to use it as a cheap-then-careful, two-stage funnel:

  1. Retrieve wide, cheaply. Vector/hybrid search grabs a generous candidate set — say the top 50 — fast.
  2. Rerank narrow, carefully. The reranker re-scores just those 50 and keeps the true best 3–5 to send onward.

You get the speed of approximate search over millions of chunks and the precision of a careful model over a small shortlist. Watch the order change once reranking is applied:

In Azure AI Search this careful second pass is offered as semantic ranking. The pattern is universal: cast a wide, fast net, then apply a precise reranker to the shortlist.

Step 4 — Augment: assemble the prompt

Now we build the actual text sent to the language model. The retrieved chunks are pasted in as context, wrapped with clear instructions. A good RAG prompt has three layers:

The prompt handed to the model
Instructions"Answer using only the context. If it's not there, say you don't know. Cite sources."
Retrieved contextTop reranked chunks + their metadata (source, page)
The question"How many days to return a faulty laptop?"
Sent to the language model

That instruction layer is doing heavy lifting. Phrases like "answer only from the context" and "if the answer isn't in the context, say you don't know" are what keep the model honest — they tell it to ground itself in your documents rather than improvise from memory.

Step 5 — Generate: a grounded, cited answer

Finally the model reads the assembled prompt and writes the reply. Because the relevant passages are right there and the instructions demand grounding, the answer is accurate and traceable:

The grounded answer
How many days do I get to return a faulty laptop?
Faulty equipment can be returned within 30 days of delivery for a full refund to the original payment method.

Source: Employee Handbook, p.12 (Refunds & Returns)

Grounded in the retrieved chunk, with a citation the user can verify — the opposite of a confident guess.

The citation comes straight from the chunk's metadata (Part 2). This is a defining advantage of RAG over a bare language model: every claim can point back to a source, which is exactly what enterprises need for trust and compliance.

How do you know it's any good? Evaluation

You can't improve what you don't measure, and "it looked fine in a demo" isn't measurement. RAG quality splits neatly into two halves — and knowing which half is failing tells you exactly what to fix:

HalfQuestion it answersIf it's weak, fix…
Retrieval qualityDid we fetch the right chunks at all?Chunking, embedding model, k, hybrid, reranking.
Generation qualityGiven good chunks, was the answer faithful and complete?The prompt/instructions, and the generation model.

The metrics you'll hear most often, in plain terms:

  • Context recall — of the chunks needed to answer, how many did retrieval actually find? Low recall means the answer wasn't even in the context.
  • Context precision — of the chunks we retrieved, how many were actually relevant? Low precision means we're feeding the model noise.
  • Faithfulness / groundedness — does the answer stick to the provided context, or did the model invent things? This is your hallucination check.
  • Answer relevance — does the response actually address what the user asked?
A practical first step: build a small set of real question-and-correct-answer pairs from your own content, then check retrieval and answers against it whenever you change anything. Microsoft Foundry and open tools like RAGAS can automate scoring once you have that set.

The takeaway — the whole series in a breath

RAG is a pipeline, not a model. Index once: load → chunk → embed → store. Answer per question: embed the query → search (hybrid + filters) → rerank → augment → generate with citations. Chunking sets the ceiling on quality; embeddings let you search by meaning; the vector index makes that search instant at scale; retrieval, reranking, and a disciplined prompt turn matches into a grounded, traceable answer. Get each box roughly right, measure both retrieval and generation, and tune from real questions — that's how a "chat with your documents" demo becomes something you can trust in production.
